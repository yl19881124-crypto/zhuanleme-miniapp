# 赚了么小程序：产品重整 PRD（V2 草案）

## 1. 背景与目标

当前版本已具备基础记账与“打工进度”能力，但存在以下问题：
- 信息架构偏散，核心路径不够聚焦（开屏后不知道先做什么）。
- 视觉和组件风格不统一，页面之间体验割裂。
- 海报能力投入高但当前业务价值低，影响迭代效率。
- 缺乏明确的版本节奏与可量化指标。

**V2 目标（4 周）**
1. 聚焦“今日状态 + 收支闭环”主流程，完成可持续日活路径。
2. 用 Stitch 统一 UI 语言与组件规范，提升一致性与可用性。
3. 暂缓海报模块，留好后续用 image2 重建的扩展位。

---

## 2. 用户与场景

### 2.1 目标用户
- 上班族，关注“今天赚了多少、花了多少、还剩多少”。
- 喜欢轻量化记录，不愿输入复杂表单。

### 2.2 核心场景
- 早上打开：确认今天上下班时间与时薪参数。
- 白天使用：快速切换状态（摸鱼/开会/加班）+ 快速记消费。
- 晚上收工：查看今日净收入、时长结构、消费构成。

---

## 3. 产品定位与核心价值

**定位**：一个“上班回血进度 + 轻记账”的日常效率小工具。  
**核心价值**：30 秒内让用户看到“今天值不值”。

---

## 4. 功能策略（MVP / 延后）

### 4.1 V2 MVP（必须做）
1. **首页（主战场）**
   - 今日已赚、净收入、进度条、下班倒计时。
   - 状态切换条（正常/摸鱼/开会/加班/收工）。
   - 快速记账浮层（金额、分类、备注可选）。
2. **记录页**
   - 今日消费时间线。
   - 支持修改/删除最近记录（至少最近 20 条）。
3. **结算页**
   - 今日战报（总收入、总支出、净收入、工作时长）。
   - 结构拆解（摸鱼时长、会议时长、有效时长）。
4. **设置页**
   - 上下班时间、时薪、午休规则、金额隐私。

### 4.2 延后（V2 不做）
- 海报生成与分享（后续 image2 重建）。
- 社交排行、好友 PK。
- 多日统计报表（先保留数据结构，不做 UI）。

---

## 5. 信息架构与导航建议

- Tab：`首页 / 记录 / 结算 / 我的`
- 非 Tab：`状态详情 / 消费编辑 / 参数设置子页 / 海报占位页`

建议将“海报”从主路径移除，仅在结算页以“即将上线”入口弱曝光。

---

## 6. UI 重做原则（Stitch）

1. **统一组件语义**：按钮、卡片、输入框、标签、弹层统一圆角/间距/字体层级。
2. **强调主任务**：首页首屏只保留 1 个核心 CTA（记一笔消费）。
3. **降低认知负担**：同屏不超过 4 个主要指标，详情放二级区块。
4. **轻反馈**：所有关键动作提供即时反馈（Toast + 数值刷新）。
5. **可扩展**：为 image2 海报区保留数据出口和视觉占位。

---

## 7. 数据与埋点（最小集）

- `home_view`
- `status_switch`
- `expense_add_success`
- `expense_edit`
- `settlement_view`
- `privacy_toggle`

关键指标：
- D1 使用完成率（当日至少完成一次“状态切换 + 记账 + 结算查看”）
- 人均日记账次数
- 结算页到达率

---

## 8. 版本节奏（建议）

### Week 1：产品与设计重整
- 完成 PRD 定稿。
- 输出 Stitch 组件清单与页面线框。

### Week 2：首页 + 记账链路重构
- 首页与弹层改版上线。
- 状态切换与实时指标稳定。

### Week 3：记录/结算/设置重构
- 完成记录编辑、结算信息结构化展示。

### Week 4：联调与灰度
- 体验打磨、性能优化、埋点核验。
- 海报入口替换为“即将上线（image2）”。

---

## 9. 是否“重新做一版”？结论

**建议：是，重做一版 V2，但不是推倒重来。**
- 复用已有：计算逻辑、存储结构、状态机。
- 重做重点：信息架构、UI 体系、关键流程交互。
- 这样可以在 4 周内把“可用”提升到“好用”，并控制风险。
